# Spike - Scraping Lewisham Website

## Context

The current repository has a FastAPI placeholder for `/bins` and an MCP wrapper that calls it. The next design decision is how the server should obtain live bin collection data from Lewisham Council's website.

This should be treated as a spike before implementation. The website appears to expose some useful JSON endpoints, but the full lookup flow is managed by Sitecore Experience Forms and client-side JavaScript. We need to confirm whether the production scraper can call stable HTTP endpoints directly, or whether it needs Playwright to drive the browser flow.

The spike also has longer-term value as drift-recovery documentation. If Lewisham changes the page, the form, or the underlying Sitecore components, a record of how the scraper was originally investigated should make it easier to re-run the same line of inquiry and realign the implementation.

## Goals

* Identify the minimum reliable flow for resolving a resident input into collection data.
* Decide whether the scraper should use direct HTTP calls, Playwright, or a hybrid approach.
* Understand the returned data shape for collection schedules.
* Define the production API interface for address lookup and bin collection schedules.
* Capture operational risks: rate limiting, cache strategy, brittle selectors, cookies, and page changes.

## Non-goals

* Building the full scraper.
* Designing every future civic-data module.
* Adding persistence, background jobs, or deployment infrastructure.
* Automating high-volume scraping against the council site.
* Designing or changing the MCP layer. MCP is an extension of this project, not part of the scraping-engine spike.

## Current observations

As of 2026-06-26, the public page is:

<https://lewisham.gov.uk/myservices/recycling-and-rubbish/your-bins/collection>

The page contains a Sitecore Experience Forms form with hidden anti-forgery/session fields and an address finder component. The visible flow asks for a postcode or street name, then lets the user select an address.

The bundled JavaScript references these endpoints:

* `POST /api/AddressFinder?postcodeOrStreet={value}&national={bool}`
* `POST /api/AddressFinder?uprn={uprn}`
* `POST /api/roundsinformation?item={item}&uprn={uprn}`

The address lookup endpoint can currently be called directly. For example, posting to `/api/AddressFinder?postcodeOrStreet={postcode_or_street}&national=False` returns JSON objects shaped like:

```json
[
  {
    "Uprn": 100000000000,
    "Title": "Example Address, Example Street, POST CODE, London"
  }
]
```

The collection endpoint still needs investigation. The JavaScript gets its `item` value from a `.js-find-collection-result` element after the form flow has progressed, but that element is not present in the initial static HTML. This suggests that the Sitecore form submission may render the result container before `/api/roundsinformation` is called.

Lewisham's `robots.txt` currently declares a sitemap and has no `Disallow` rules.

## Key questions

1. Can collection data be retrieved with direct HTTP requests once we know the correct `item` value?
2. Is the `item` value stable across sessions and deploys, or generated by Sitecore per form render?
3. What is the exact response schema from `/api/roundsinformation`?
4. Does the result vary by property type, flats, commercial properties, or garden-waste subscription state?
5. Does the flow require cookies, anti-forgery tokens, or Sitecore form session fields for collection results?
6. What failure modes does the site expose for invalid postcodes, ambiguous streets, no results, and non-Lewisham addresses?
7. Are collection changes, bank holidays, and service alerts included in the result payload or only rendered elsewhere on the page?
8. Do public/civic buildings return collection schedules, or does the schedule endpoint only cover domestic residential UPRNs?

## Proposed investigation

### 1. Trace the browser flow

Use Playwright in headed or trace mode against the live page with a small number of known Lewisham test inputs.

Capture:

* Network requests and responses.
* Cookies and required headers.
* Form posts to `/formbuilder`.
* Any request to `/api/roundsinformation`.
* The DOM before and after address selection.
* The final rendered collection result.

Suggested test inputs:

* `1, Abbotshall Road, SE6 1SQ, London`, for residential collection-schedule validation.
* One invalid or non-Lewisham postcode.

Ideally, this spike would use an anonymous public building as the stable fixture. In practice, a small manual sample of public or public-facing buildings did not expose regular refuse or recycling collection schedules through Lewisham's public bin collection page. The buildings checked were:

* Lewisham Council Office.
* Catford Library.
* Fox and Firkin.
* Forest Hill Library.
* The Albany.

From this, the working conclusion is that a residential address is required to validate the domestic collection-schedule flow. Public buildings can still be useful for address-finder smoke testing, but they should not be treated as viable end-to-end collection fixtures unless a future check proves otherwise.

The residential test address was selected systematically rather than personally: after checking the Lewisham highway register, Abbotshall Road was the first road in an A-Z ordering, and number 1 was the first address returned by the address finder for its postcode. This gives the spike a stable domestic fixture without targeting a known resident, contributor, or personally meaningful property.

Do not commit residential UPRNs to the repository. If additional residential addresses are needed during local testing, pass them through local environment variables or a gitignored scratch file, document only the response shape, and anonymize any fixture.

If a public civic building resolves addresses but returns no collection result, treat that as a valid finding rather than a scraper failure.

### Sitecore-specific checks

During the trace, pay particular attention to the `item` parameter in `/api/roundsinformation?item={item}&uprn={uprn}`.

In Sitecore, this is likely to be a Sitecore item ID for the component or content item that owns the bin collection result. It may be stable across user sessions, but it should not be assumed to be permanent. If a content editor moves, replaces, or republishes the component, the ID could change.

The spike should therefore answer:

* Is `item` injected into the DOM, for example as a data attribute on `.js-find-collection-result`?
* Is `item` visible only after the Sitecore form has been submitted?
* Can the production scraper extract `item` dynamically from the page or form response?
* Does `/api/roundsinformation` work without a prior `/formbuilder` POST?

If the collection endpoint returns `403`, an empty payload, or inconsistent data under `httpx`, assume the endpoint may depend on browser-managed state. Sitecore Experience Forms can rely on anti-forgery tokens, transient session cookies, and a form-state progression that proves the user moved through the intended steps.

### 2. Replay the flow with HTTP only

Once the Playwright trace identifies the exact request sequence, try to reproduce it using `httpx`.

Start with:

1. Fetch the collection page.
2. Extract hidden Sitecore form fields if needed.
3. Call `/api/AddressFinder` for postcode/street lookup.
4. Select a UPRN.
5. Submit any required Sitecore form step.
6. Call `/api/roundsinformation`.

If this works consistently, prefer direct HTTP for the production scraper. It will be faster, easier to test, cheaper to run in containers, and less fragile than controlling a full browser.

When replaying with `httpx`, use a browser-like request profile rather than the default client fingerprint. At minimum, set a clear project `User-Agent`, normal `Accept` headers, and `Accept-Language`. If Lewisham's hosting stack or web application firewall treats bare automation clients differently, the spike should document that behavior rather than working around it silently.

### 3. Decide whether Playwright remains necessary

Use Playwright in production only if one of these is true:

* The result endpoint cannot be reached without browser-managed state.
* Required tokens are produced by browser-executed JavaScript.
* The form flow depends on browser behavior that is expensive or brittle to recreate.
* The collection result is only available as rendered HTML and not as JSON.

If Playwright is needed, keep its responsibility narrow: resolve the council flow and return normalized data to the application. The FastAPI router should not know whether the data came from Playwright or direct HTTP.

## Candidate architecture

The server package should keep the HTTP API surface, business orchestration, outbound source clients, state gateways, and pure domain objects easy to distinguish. One possible structure is:

```text
lewisham_server/
  api/
    routers/
      bins.py
    schemas/
      bins.py
  services/
    bins_service.py
  clients/
    lewisham/
      client.py
      parser.py
  storage/
    cache_interface.py
    memory_cache.py
  domain/
    models.py
    errors.py
```

Likely responsibilities:

* `api/routers/bins.py`: FastAPI route declarations for the bins feature, including address lookup and collection-schedule routes under the `/bins` namespace.
* `api/schemas/bins.py`: Pydantic response shapes for bin-related API responses, including address candidates and collection schedules.
* `services/bins_service.py`: business orchestration for the bins use case, including address discovery, schedule lookup, cache policy, and domain errors.
* `clients/lewisham/client.py`: outbound Lewisham gateway, responsible for direct HTTP and, if needed, Playwright-backed transport.
* `clients/lewisham/parser.py`: source-specific parsing and normalization of raw Lewisham/Sitecore responses into domain models.
* `storage/cache_interface.py`: abstract cache contract used by the service layer.
* `storage/memory_cache.py`: in-memory cache implementation for local development and early production use.
* `domain/models.py`: pure internal representations of addresses, UPRNs, collection schedules, and source metadata.
* `domain/errors.py`: application-specific exceptions such as `AddressNotFoundError`, `CollectionScheduleNotFoundError`, or `UpstreamScraperChangedError`.

This separates API schemas from domain models. API schemas can evolve around client serialization, status fields, and HTTP presentation, while domain models stay focused on the internal truth the service layer works with.

Address lookup is included under the bins API because, for now, it is part of the bin collection journey and is discovered through Lewisham's bin page. If address lookup later becomes a shared civic-data primitive for planning, roads, or other modules, it can be promoted to its own top-level API surface then.

The parser belongs with the Lewisham client because it is source-specific. It exists because Lewisham's website may return awkward JSON, form responses, or HTML. If another council or civic data source is added later, it can get its own client and parser without changing the bins route contract.

The storage interface should stay small until there is a real second implementation. It is useful as a boundary because cache policy is central to responsible scraping, but it should not grow into a generic repository layer before the project needs one.

The MCP package is deliberately out of scope for this spike. The scraping engine should be designed around the REST API and domain boundary first; MCP can adapt to that later if it remains useful.

## Proposed API shape

The API should separate address discovery from collection-schedule retrieval. These are different resources with different schemas, so they should not be multiplexed through a single `/bins` endpoint.

The routes can still live under the `/bins` namespace because the address lookup is part of the bin collection flow and the current source is Lewisham's bin collection page.

Address lookup:

```http
GET /bins/addresses?postcode={postcode}
```

This endpoint always returns an array. A postcode with one matching property returns a one-item array; an ambiguous postcode returns multiple candidates; a postcode with no Lewisham matches returns an empty array.

```json
[
  {
    "uprn": "100000000000",
    "title": "Example Address, Example Street, POST CODE, London"
  }
]
```

Collection schedule lookup:

```http
GET /bins/addresses/{uprn}/collections
```

This endpoint always returns a single bin collection schedule for a known property UPRN, or an error if the property cannot be resolved or has no public domestic collection schedule.

```json
{
  "uprn": "100000000000",
  "address": "Example Address, Example Street, POST CODE, London",
  "collections": [
    {
      "type": "rubbish",
      "date": "2026-06-30",
      "status": "scheduled"
    }
  ],
  "source": {
    "url": "https://lewisham.gov.uk/myservices/recycling-and-rubbish/your-bins/collection",
    "fetched_at": "2026-06-26T00:00:00Z"
  }
}
```

This nested shape treats bin collections as a sub-resource of an address/property within the bins feature.

If address lookup later becomes a shared top-level civic primitive, the API could grow toward:

```http
GET /addresses/{uprn}/bin-collections
GET /addresses/{uprn}/planning-applications
GET /addresses/{uprn}/road-closures
```

A flatter bins-scoped shape would also be workable:

```http
GET /bins/addresses?postcode={postcode}
GET /bins/collections/{uprn}
```

The nested bins-scoped form is preferred for now because it is deterministic while staying honest about the current feature boundary.

The exact collection fields should wait until the `/api/roundsinformation` response is observed.

## Caching and load policy

The scraper should be deliberately conservative:

* Cache successful UPRN lookups for at least 24 hours.
* Cache address lookup responses for a shorter period, for example 7 days, because UPRNs and addresses change rarely.
* Cache negative responses briefly, for example 1 hour.
* Add a clear `User-Agent` identifying this project if direct HTTP requests are used.
* Add request timeouts and bounded retries.
* Avoid background crawling; fetch only for user-requested postcodes or UPRNs.

## Scraping principles

This project should keep good scraping practice at the centre of the implementation:

* Prefer the smallest reliable request path that produces the needed public data.
* Do not scrape more addresses than the user requested.
* Do not use Playwright in production unless the site flow genuinely requires browser execution.
* Treat private addresses, UPRNs, and raw traces as sensitive operational data.
* Keep fixtures anonymized unless the fixture is intentionally public and non-personal.
* Document source-site assumptions so future page drift can be diagnosed without rediscovering everything from scratch.
* Fail clearly when the upstream page changes, rather than returning guessed or stale collection data as if it were authoritative.

## Risks

* **Sitecore form fragility:** hidden fields or form actions may change.
* **Sitecore item drift:** the `item` parameter may refer to a component or content item that changes when the page is edited.
* **Session coupling:** `/api/roundsinformation` may depend on cookies, anti-forgery tokens, or successful form-state progression.
* **Undocumented endpoints:** `/api/AddressFinder` and `/api/roundsinformation` are not public contracts.
* **WAF or bot filtering:** default automation clients may receive different responses from browser-like traffic.
* **Result schema drift:** collection payload fields may change without notice.
* **Address ambiguity:** one postcode can return many UPRNs.
* **Sensitive user input:** addresses and UPRNs should be treated as personal data in logs.
* **Browser dependency:** Playwright would add image size, install complexity, and runtime cost.

## Spike deliverables

The spike should end with:

* A short trace summary of the live website flow.
* A captured example of the `/api/roundsinformation` response from a residential UPRN, anonymized before committing.
* A note on whether public/civic UPRNs return no collection data.
* A note on where the Sitecore `item` value comes from and whether it can be extracted dynamically.
* A note on whether collection lookup is stateless or depends on browser/session/form state.
* A decision: direct HTTP, Playwright, or hybrid.
* A thin proof-of-concept script, either committed under a scratch/spike location or summarized and discarded.
* A recommended implementation ticket list for the production scraper.

## Tentative recommendation

Start the spike with Playwright, but do not assume Playwright is the production answer.

Playwright is the best discovery tool here because it can reveal the true browser flow, including form submissions and hidden network requests. If the trace proves that the useful endpoints are callable with ordinary HTTP, the production scraper should use `httpx` and structured parsing instead. That gives us a smaller, more testable, and more deployable service while keeping Playwright available as an investigation tool or fallback.
